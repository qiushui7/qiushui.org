# Repository Guidelines

## Project Structure & Module Organization
- `src/app/`: Next.js App Router routes (pages, layouts, API routes).
- `src/components/`: reusable UI components (React + Tailwind).
- `src/lib/`: shared utilities and data access (see `src/lib/db/schema.ts`).
- `src/blog/`: file-based MDX posts: `src/blog/<category>/<slug>.mdx`.
- `public/`: static assets served as-is.
- `supabase/migrations/`: SQL migrations generated by Drizzle (treat as generated output).

## Build, Test, and Development Commands
Use the exact `pnpm` version pinned in `packageManager` (currently pnpm 11 via Corepack). Do not use npm or Yarn to update dependencies.
- `pnpm install --frozen-lockfile`: reproduce the committed dependency tree.
- `pnpm install`: update `pnpm-lock.yaml` after an intentional manifest or pnpm policy change.
- `pnpm dev`: run local dev server (Turbopack) at `http://localhost:3000`.
- `pnpm build` / `pnpm start`: production build and serve.
- `pnpm lint`: run ESLint (Sukka formatter).
- `pnpm type-check`: run TypeScript checks (`tsc --noEmit`).
- `pnpm audit --prod` / `pnpm audit`: audit production dependencies and the complete dependency tree.
- `pnpm db:gen`: generate migrations from `src/lib/db/schema.ts`.
- `pnpm db:mig`: apply migrations (requires `POSTGRES_URL`).

## Coding Style & Naming Conventions
- TypeScript strict mode; prefer `@/*` imports (maps to `src/*`).
- Components: `PascalCase.tsx`; hooks: `useSomething.ts`; utilities: `kebab-case.ts` or match existing file patterns.
- MDX slugs and category folders: `kebab-case` (e.g., `reading-notes/code-art.mdx`).
- Run `pnpm lint` before opening a PR; formatting is enforced via ESLint (no separate Prettier step).

## Testing Guidelines
No dedicated test runner is configured in this repo today. Validate changes with:
- `pnpm lint` and `pnpm type-check` for all code changes.
- `pnpm build` for framework, dependency, configuration, routing, MDX, or server-side changes.
- `pnpm audit --prod` and `pnpm audit` after dependency or lockfile changes; do not leave known vulnerabilities without documenting why they cannot be fixed.
- a quick manual smoke test via `pnpm dev` for affected routes.

## Dependency and Supply-Chain Management
- Keep `package.json`, `pnpm-lock.yaml`, and `pnpm-workspace.yaml` in sync; never hand-edit the generated lockfile.
- `pnpm-workspace.yaml` contains security overrides and the reviewed dependency build-script policy. Prefer narrow affected-version overrides and exact-version `allowBuilds` entries.
- Inspect a package's install script before allowing it. Do not broadly allow all dependency scripts or weaken the existing explicit denial for `@vercel/speed-insights`.
- Keep the registry URL HTTPS-only. The repository uses the official npm registry configured in `.npmrc`.
- Keep build and migration CLIs such as `drizzle-kit` in `devDependencies` unless application runtime code imports them.
- Remove an override only after the dependency graph resolves to a patched version and both audit commands remain clean.

## Commit & Pull Request Guidelines
- Commit messages follow Conventional Commits (common types: `feat:`, `fix:`, `chore:`, `perf:`, `style:`).
- PRs should include: a short summary, screenshots for UI changes, and migration files when `schema.ts` changes.

## Security & Configuration Tips
- Do not commit secrets: `.env*` is ignored; use local environment variables (notably `POSTGRES_URL`).
- Never pass request-controlled values directly to Drizzle identifier or alias APIs such as `sql.identifier()` or `.as()`; map dynamic choices through an allowlist of static schema objects.
- Treat MDX and frontmatter as executable/build input. Only compile trusted repository content unless an explicit sanitization and sandboxing design is added.
